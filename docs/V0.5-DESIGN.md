# v0.5 Working Design and Delivery Plan

- **Status:** working design; non-normative
- **Recorded:** 2026-07-20
- **Behavioral baseline:** [`v0.4.0`](https://github.com/MosslandOpenDevs/agentic-assurance-profile/releases/tag/v0.4.0), commit [`00e2fe46d4eb01a4147f149851a48a3017cbb796`](https://github.com/MosslandOpenDevs/agentic-assurance-profile/commit/00e2fe46d4eb01a4147f149851a48a3017cbb796)
- **Development baseline when recorded:** [`ff171d3d282e9f896c10851270ee3918d89c64fe`](https://github.com/MosslandOpenDevs/agentic-assurance-profile/commit/ff171d3d282e9f896c10851270ee3918d89c64fe), `VERSION=v0.4.1-dev`

This document records a proposed product, architecture, compatibility, UX,
testing, and issue plan for v0.5. It does not change an adopter obligation.
[PROFILE.md](../PROFILE.md) remains the only normative profile text.

The command name `aap` is provisional throughout this document. A public CLI
or package MUST NOT be published until the namespace decision in §14 is closed.

## 1. Executive decision

v0.5 should reduce the surface that adopters and maintainers must understand
while preserving the depth and honesty of the assurance model.

It should **not** begin with a full engine rewrite. It should begin with a
mandatory, independently useful foundation:

1. freeze the immutable v0.4.0 behavior corpus;
2. measure v0.4.0 adoption rather than infer its usability from repository CI;
3. give findings stable identities and a versioned machine contract;
4. extract the trusted post-checkout workflow application from YAML;
5. make check applicability explicit so a skipped check cannot look green;
6. put a thin CLI facade over the existing engine.

Only then does the project take a go/no-go decision on a canonical Snapshot
engine. A no-go is a valid outcome: the foundation above is enough to make a
useful v0.5 release without freezing an untested domain abstraction.

If the gate passes, the new engine replaces the old one by a strangler pattern:
loaders and typed findings first, snapshot rules second, transition rules last,
with the v0.4.0 corpus acting as a frozen oracle throughout.

1. The 428 tests are not generally coupled to private function structure.

### 2.3 INFERRED: architecture pressure

The same concepts are normalized repeatedly across lite/split layout, default
paths, implicit `core`, active/archived behavior, snapshot validation, and
base/head transitions. A typed boundary is likely to reduce this duplication
and make structured diagnostics, migration, and isolated Git snapshots safer.

This inference is strong enough to justify the foundation work. It is not yet
strong enough to justify making every proposed canonical type a public or
permanent compatibility contract.

### 2.4 UNKNOWN: adopter usability

The repository does not yet establish:

- how long an unfamiliar adopter needs to reach an owner-reviewable draft;
- whether the minimal lite route feels minimal in practice;
- which terms or decisions cause the most confusion;
- whether users get trapped in agent fix/fail loops;
- how many warnings are false or non-actionable;
- whether the dominant pain is authoring, classification, review, CI setup, or
  engine diagnostics;
- whether more than one real adopter needs the same new abstraction.

These are measurements for Phase 0, not assumptions that the rewrite may use
as premises.

## 3. Product definition

Working product sentence:

> Repository-native assurance for AI-assisted software change: check whether
> the promises a human approved, the controls and evidence supporting them,
> and the known risk that human accepted remain represented after change.

The durable differentiation is not that no neighboring tool can ever cover
the same territory. It is a small, deterministic semantic transition engine
that can consume existing specs, tests, scanners, review results, provenance,
and attestations without pretending those outputs prove more than they do.

### 3.1 v0.5 goals

- One obvious read-only entry point for the common check.
- A preview-first path that creates a useful DRAFT without fabricating human
  authority or evidence.
- Stable finding identity for CI, agents, documentation, and future adapters.
- Explicit `applicable`, `not_applicable`, and `not_run` check states.
- An internal model that can represent lite and split artifacts consistently
  while preserving every physical source and revision.
- A safe compatibility and migration boundary instead of permanent historical
  special cases throughout current rules.
- A reusable workflow whose YAML contains only the necessary trust bootstrap,
  checkout, dependency setup, and module entry points.
- Deterministic, offline, resource-bounded, fail-closed behavior.

### 3.2 Anti-goals

v0.5 is not:

- a new normative assurance ontology;
- a certification, safety claim, bug-free claim, or security scanner;
- a spec generator, task manager, coding-agent orchestrator, or session recorder;
- a code graph, taint engine, generic SAST platform, sandbox, or runtime agent
  authorization system;
- a hosted dashboard or a large vendor-specific adapter collection;
- an automatic judge of human intent, truth, profile classification, residual
  acceptance, or approval;
- a reason to expose a full internal Snapshot, restricted policy content, or
  raw sensitive values in CI JSON;
- a requirement to eliminate every line of workflow bootstrap code.

## 4. What v0.5 preserves and what it may replace

The v0.4 hardening must be classified before code moves.

| Preserve independent of engine | Candidate for structural absorption |
|---|---|
| Bounded YAML/JSON/prose reads and node/depth/token limits | Repeated lite/split projection branches |
| Duplicate-key, merge-key, cyclic, non-finite, and malformed-input rejection | Repeated missing/`None`/list state conventions |
| Repository-relative path, containment, symlink, and trust-root checks | Repeated declared/effective profile derivation |
| Exact pin, checked-out revision, clean-resource, and canonical-tag checks | Compatibility fingerprints scattered through active rules |
| Injection-safe logs, annotations, and JSON; controlled diagnostics | Renderer strings used as rule identity |
| Stable IDs, stage semantics, profile obligations, and base/head protections | Workflow reimplementations of loaders and transition preparation |
| Fail-closed behavior when an applicable check cannot complete | Raw dictionaries crossing parser, semantic, and transition boundaries |

The goal is not fewer defensive tests. The goal is to express each defense
once at the correct boundary and stop multiplying states that later rules must
handle.

## 5. Delivery sequence

### Phase 0 — observe and close false-green ambiguity

This phase starts from immutable v0.4.0 behavior, even though development
branches start from current `main`.

Deliverables:

1. Record a fixture manifest covering active/archived, lite/split/mapped,
   current/legacy inputs, stage outcomes, transition regressions, path and
   trust failures, drift routing, and central self-check.
2. Add a real dogfood adopter fixture; do not rename central self-check as
   self-adoption.
3. Run at least two end-to-end v0.4.0 adoption trials: one existing brownfield
   adopter and one repository not constructed as an engine fixture. At least
   one trial SHOULD be performed or observed by someone other than the primary
   engine implementer.
4. Record the usability metrics in §12 and classify each problem as content,
   classification, authoring, review, CI integration, diagnostic, or engine
   structure.
5. Resolve #43's event contract before the engine rewrite: every check reports
   applicability and completion separately.

For #43, a v0.4.1 patch is appropriate only if the fix can make the existing
event contract explicit without adding a privileged execution path or breaking
supported callers. If safe handling requires new event permissions, new
base/head semantics, or `pull_request_target`, it belongs in the v0.5 CI
adapter after a threat-model review. PR-controlled code MUST NOT execute in a
privileged `pull_request_target` context.

### Phase 1 / alpha.1 — contracts before internals

This work is required regardless of the canonical-engine decision.

1. Introduce immutable `Finding`, `SourceRef`, check-status, and gate-decision
   objects behind the current rules.
2. Allocate stable finding codes and publish a legacy-message-to-code mapping.
3. Add JSON diagnostic schema v1 and a human text renderer.
4. Put a thin `aap check` facade over v0.4 behavior while retaining
   `scripts/validate.py` as a compatibility wrapper for the v0.5 line.
5. Extract post-trusted-checkout pin reading, change normalization, and the
   base/head materializer into package modules with direct tests.
6. Keep the pre-checkout identity check and checkout-destination reservation
   in YAML. There is no trusted local module before the fixed source checkout.
7. Preserve isolated execution (`python -I`), explicit trusted import paths,
   hash-locked dependencies, and adopter-import-shadowing defenses.

At the end of alpha.1, finding messages may improve, but code, rule identity,
gate effect, source, and check completion become the comparison contract.

### Gate G1 — canonical engine go/no-go

Proceed to Phase 2 only when all of the following are true:

- **Oracle ready:** every supported v0.4 behavior class has expected structured
  outcomes; unexplained code/level/gate differences are zero.
- **Pilot evidence ready:** the two adoption trials have complete notes and at
  least one repeated problem cannot be solved only by documentation or a CLI
  facade, or a required feature such as isolated staged snapshots or migration
  demonstrably duplicates current normalization.
- **Spike ready:** a bounded prototype can normalize lite/split and
  `[service]`/`[core, service]` without losing physical source, revision, or
  trust information.
- **Scope clean:** no new normative obligation is mixed into the engine move.
- **Fallback viable:** the project can still release the Phase 1 foundation if
  the prototype is stopped.

If any condition fails, stop the full rewrite. Ship the boundary, finding, CI,
and CLI work as v0.5.0, collect more adoption evidence, and revisit the engine
in a later minor. This is not a failed v0.5.

### Phase 2 / alpha.2 — canonical model behind the old engine

Introduce the model as an internal seam, not a new public file format:

```text
RawDocument
  -> bounded ParsedDocument
  -> explicitly versioned MigratedDocument
  -> PartialSnapshot
  -> CanonicalSnapshot
  -> snapshot and transition rules
  -> Finding
  -> GateDecision
  -> text / JSON / GitHub adapters
```

Every stage returns an outcome with findings and completion state. Malformed or
unsafe input need not produce a CanonicalSnapshot; it must produce a controlled
failure and mark dependent checks `not_run`, never silently pass them.

The minimum domain model includes:

- `AdoptionMode = Active | Archived`;
- physical `Layout = Lite | Split | Mapped`;
- both `declared_profiles` and spec-derived `effective_profiles`;
- adoption stage, profile pin, artifact-format version, and register graph;
- base/head revision identities for transitions;
- `SourceRef(uri, pointer, line, column, revision)` where known;
- `Origin = EXPLICIT | DEFAULTED | INHERITED | MIGRATED | SYNTHESIZED` so a
  derived value never masquerades as adopter-authored source.

Unknown line or column is `null`; the engine never invents a location. A
normalized value retains its physical source so path relocation, symlink,
review-boundary, and base/head checks remain possible.

### Phase 3 / beta.1 — move rules under parity

Move rule families in this order:

1. parsing, schema, and bounded repository reads;
2. profile composition, required artifacts, and layout normalization;
3. snapshot semantics and stage rules;
4. component routing and change facts;
5. adoption, path, and register transition regressions.

For each family, run old and new engines on the same inputs. Compare structured
semantic results, not full prose formatting. The old implementation becomes a
frozen oracle and receives no new feature rules.

Only intended differences with a reviewed migration note may enter the parity
allowlist. An allowlist entry states the old outcome, new outcome, normative
reason, affected fixtures, and compatibility impact.

### Phase 4 / beta.2 — progressive UX

After the diagnostic and snapshot contracts are trustworthy, add the user
surface in §8. The normal route is `init -> check -> review`; advanced
`migrate` and `explain` commands do not make the common path larger.

This phase runs the same adoption trials again and compares the measured
results with Phase 0. It does not claim better usability merely because the
command is shorter.

### Phase 5 / rc.1 and final

Before rc.1:

- freeze the finding catalog and JSON protocol for v0.5;
- freeze new scope and classify every remaining report as blocker, documented
  limitation, or later issue;
- make the new engine the default only if Phase 3 parity is complete;
- when the new engine becomes the default, retain the old engine only as a
  test oracle and compatibility wrapper; otherwise keep the Phase 1 engine as
  the supported implementation;
- publish migration notes, old-message-to-code mapping, event semantics, and
  exact package/profile pin rules;
- rerun the two pilot adoptions on the release candidate.

Final follows rc.1 only when the release blockers in §13 are closed. New
ontology, generic evidence envelopes, SARIF export, and vendor adapters are not
allowed to reopen the release scope.

## 6. Architecture boundaries

The target dependency direction is:

```text
bounded document and repository adapters
              |
              v
     partial/canonical snapshots
              |
              v
   snapshot rules -> transition rules
              |
              v
          typed findings
              |
              v
        operational gate policy
              |
              v
 text / JSON / GitHub / future SARIF renderers
```

Rules do not read arbitrary files or raw YAML. Renderers do not decide policy.
Gate policy may change severity or blocking effect according to an explicit
stage or CI mode, but it does not change finding identity.

A practical initial package split is:

| Boundary | Responsibility |
|---|---|
| `finding` | Finding, SourceRef, check status, redaction, deterministic ordering, renderers |
| `policy_io` | bounded YAML/JSON/prose parsing, schema registry, source marks |
| `snapshot` | declared/effective profiles, defaults, lite/split projection, domain values |
| `rules` | snapshot, semantic, and stage rules |
| `transition` | adoption/path/register regressions and base/head comparison |
| `repo` | containment, symlink, Git object, revision, and trust-boundary reads |
| `ci` | pin verification, event facts, base/head materialization, annotations and summary |
| `commands` | self-check, check, review preparation, migration, compatibility entry points |

These names are illustrative. The architectural test is that snapshot and
transition rules cannot reach back into raw YAML or the ambient filesystem,
not that the repository contains a particular number of files.

PROFILE.md stays normative. A mechanical rule catalog may map rule ID,
applicability, required artifact, transition kind, and finding code, but it
must not silently replace or generate the normative prose.

## 7. Finding and gate contract

A finding code identifies a semantic condition. It does not encode severity,
stage, renderer, or the actor believed able to fix it.

Minimum finding data:

- stable `code` and normative `rule_id`/reference;
- category and title;
- message or message facts, with untrusted values bounded and escaped;
- one or more source locations with role and revision;
- default severity, evaluated severity, and gate effect as separate values;
- required authority and automation policy for the next action;
- deterministic ordering and redaction metadata.

Minimum run data:

- tool, profile release, profile commit, and diagnostic-format versions;
- evaluated base/head identities or current snapshot kind;
- every required check's applicability, completion status, and outcome;
- findings, summary, overall outcome, and exit code.

The public JSON report MUST NOT expose the full internal CanonicalSnapshot.
That would prematurely freeze the private model and could leak restricted or
embargoed content through CI artifacts. JSON is an AAP diagnostic projection;
SARIF, if added later, is another intentionally lossy projection.

Text may use readable sections such as `Required`, `Observed`, and `Next step`,
but text layout and wording are not the machine API. A diagnostic may omit a
section that does not fit. A next step may prepare or request a decision; it
must not grant authority.

### 7.1 Exit status

The new CLI uses outcome classes, not guessed remediation actors:

| Exit | Meaning |
|---:|---|
| `0` | Evaluation completed and the gate passed; warnings may exist. |
| `1` | Evaluation completed and one or more findings block the gate. |
| `2` | CLI usage or option-combination error. |
| `3` | A required evaluation could not complete safely: dependency, I/O, Git snapshot, unsupported applicable event, or internal failure. |
| `128 + signal` | Conventional signal termination. |

Exit `3` means indeterminate, not “human approval required.” A run containing
both policy findings and an incomplete required check returns `3`, because it
cannot make a complete gate claim. Malformed policy that was safely parsed as
an evaluated schema/semantic failure normally returns `1`; inability to read or
evaluate the required policy returns `3`.

Whether an agent may propose a mechanical change or must request human-owner
judgment belongs to each finding's `authority_required`, `automation`, and
`auto_apply` fields. It is never inferred from the process exit code.

### 7.2 Output channels

- `stdout` contains exactly the selected result: human text or one UTF-8 JSON
  document.
- `stderr` contains usage, bootstrap failure, and opted-in diagnostics.
- JSON never contains ANSI color, progress logs, or GitHub workflow commands.
- GitHub annotations are produced by an explicit CI adapter, not automatically
  because `GITHUB_ACTIONS` exists.
- `--quiet` applies only to text and suppresses successful/non-blocking output.
- text honors `NO_COLOR` and `--color auto|always|never`.

The bootstrap must determine output mode using the standard library before
optional dependency imports, so a missing dependency cannot corrupt requested
JSON with an unstructured line.

## 8. CLI and human-authority UX

### 8.1 `aap check`

Default behavior is read-only, deterministic, network-free, and repository
wide. It discovers the adoption declaration and pinned profile context instead
of requiring users to pass schema and checkout paths.

Initial options:

- `--format text|json`;
- `--quiet` and `--color` for text;
- `--base <ref> --head <ref>` only after Git snapshot isolation exists;
- `--staged` only when base=`HEAD` and head=`index` reads every policy artifact
  from the index, never the mixed worktree;
- `aap explain <finding-code>` for rule help.

The CLI does not implicitly fetch. Missing refs, shallow history that prevents
the required comparison, a conflicted index, or an unsafe snapshot is exit `3`.

Do not add `--profile <path>` as a gate-narrowing option. `profile` already
means `core/service/...`, and partial validation could suppress an obligation.
A future `--focus` may filter display while still evaluating the complete gate;
any genuinely partial result must be labeled partial and cannot report
repository conformance.

### 8.2 `aap init`

`init` lowers authoring cost without claiming to know the adopter's truth.

- Preview is the default; filesystem changes require `--apply`.
- It scans for profile-trigger evidence and proposes candidate classification
  with `file:line` support, biasing ambiguous cases toward owner review.
- It may create a DRAFT with `UNKNOWN` conclusions and unresolved decision
  prompts.
- It never auto-creates `VERIFIED`, accepts a residual, closes a defeater,
  promotes a stage, or asserts a human decision.
- The declared profiles in a generated DRAFT remain a proposal until the
  repository's durable owner-review process confirms them.
- The tool records what it defaulted, inherited, migrated, or synthesized.

[#41](https://github.com/MosslandOpenDevs/agentic-assurance-profile/issues/41)
may later improve this proposal layer, but a heuristic never becomes the
canonical effective profile. Human-approved declared profiles remain the policy
input; code signals are advisory evidence and possible escalation.

### 8.3 `aap review`

`review` is read-only by default. It produces an owner review packet containing
unresolved decisions, affected stable IDs, cited evidence, transition findings,
and concrete questions.

`aap review prepare --write-draft` may write a clearly non-authoritative draft
when explicitly requested. There is no `aap review --approve` command. A local
TTY, username, confirmation prompt, or agent invocation is not evidence of
human authority. The owner's actual review/approval in the repository's durable
process creates the authority; tooling may later verify and record that event,
but not manufacture it.

### 8.4 `aap migrate`

Migration previews by default and changes files only with `--apply`.

It preserves stable IDs and represented semantics. It may make a mechanical
format change, but it never invents intent, evidence, classification, review,
status, defeater disposition, residual acceptance, or risk truth. Any such
missing decision is emitted as a review item.

## 9. Versioning and compatibility

Version these contracts separately:

1. **profile release** — normative profile and bundled implementation release;
2. **artifact format** — on-disk adoption/register representation;
3. **finding protocol** — JSON report and stable diagnostic semantics;
4. **evidence envelope** — a future integration contract, not part of v0.5
   merely because MAPPINGS §5 exists.

The adoption format must stop inferring all compatibility from
`upstream.version`. The exact field placement and initial version are an ADR,
but v0.5 must have an explicit artifact-format identity.

Compatibility rules:

- immutable v0.4.0 adopters continue using the pinned v0.4.0 workflow/validator;
- `scripts/validate.py` remains a v0.5 compatibility wrapper;
- v0.5 text wording may change, but a published mapping explains every legacy
  condition's new finding code;
- `aap check` initially supports current and immediately previous artifact
  formats; older formats route through explicit chained migration;
- a v0.5 normative local check requires an engine/profile release match rather
  than silently evaluating a pinned profile with “latest” semantics;
- the reusable CI workflow continues executing the exact full-SHA trusted
  checkout, which is the authoritative way to avoid an installed-tool mismatch;
- migration never silently upgrades a repository's human-approved meaning.

## 10. Test strategy

### 10.1 Keep the 428-test behavior asset

Classify existing tests instead of replacing them wholesale:

- **behavioral/characterization:** supported input, exit, semantic finding, and
  compatibility behavior; survives the rewrite;
- **trust/integration:** Git, path, symlink, resource, injection, and subprocess
  boundaries; remains integration-heavy;
- **workflow structural:** exact YAML step names, indentation, shell, and inline
  body; intentionally rewritten as module tests plus a small wiring contract;
- **renderer compatibility:** a small legacy text subset and the new stable
  structured contract.

The v0.4 text corpus first receives stable-code mappings. New parity asserts
check status, finding code, evaluated severity, gate effect, relevant source,
and exit class. It does not require byte-for-byte preservation of every human
sentence.

### 10.2 Add focused unit and property tests

High-value properties are:

- normalization is idempotent;
- lite and equivalent split inputs produce equal semantic snapshots while
  retaining different SourceRefs;
- `[service]` and `[core, service]` have equal effective obligations while
  preserving different declarations;
- migration preserves stable IDs and semantic values and never upgrades human
  decisions;
- unchanged base/head snapshots create no transition regression;
- finding order and JSON output are deterministic;
- malformed or over-budget input never yields a dependent check pass;
- redaction and escaping prevent terminal, JSON, Markdown, and GitHub-command
  injection;
- index, worktree, and commit snapshots never mix content.

Central self-check remains a first-class fixture. The separate dogfood adopter
fixture becomes the first-class self-application parity case.

## 12. Usability and engineering evidence

Record these for the v0.4 baseline and v0.5 beta trials:

| Measure | Why it matters |
|---|---|
| Time to owner-reviewable DRAFT | Separates scaffold speed from the misleading goal of instant green. |
| Time from DRAFT to first trustworthy passing check | Captures the full adoption path. |
| Files and YAML fields manually authored | Tests whether the CLI hides representation work. |
| Profile trigger decisions and unresolved human decisions | Preserves necessary authority instead of hiding it. |
| Terminology lookups or clarification requests | Identifies ontology UX pressure. |
| False/non-actionable warnings | Tests #41 and diagnostic quality. |
| Agent fix/fail loop count | Tests whether authority and next steps are clear. |
| Review-packet questions answered or rewritten by the owner | Measures whether review output is useful. |
| Check runtime and variance on the same repository | Detects performance regressions without inventing a universal threshold. |
| Finding-code churn across alpha/beta | Tests whether the diagnostic catalog was frozen too early. |
| Change amplification per rule family | Tests whether the architecture reduces repeated edits. |

For a minimal `core` adoption, target no direct YAML authoring when `init` is
used and no more than the existing essential artifact surface. Do not minimize
the number of genuine human decisions: surfacing those decisions is a product
function. Minimize repeated entry and ambiguity around them.

The G1 report records observations, raw counts, repository/revision, operator,
and limitations. It must distinguish `VERIFIED`, `INFERRED`, and `UNKNOWN`
rather than converting two pilots into a market-wide claim.

## 13. Release blockers

The following block rc or final:

- a trust-boundary, resource-bound, injection, or fail-closed regression;
- an applicable required check that is skipped, reported not-applicable, or
  rendered as pass;
- an unexplained v0.4 corpus difference in finding code, evaluated severity,
  gate effect, or protected semantic transition;
- nondeterministic or non-schema-valid JSON;
- default JSON/text output that leaks restricted content or raw sensitive values;
- migration that changes a stable ID or human decision without explicit review;
- a staged/base/head check that mixes snapshots;
- an installed/pinned engine mismatch that proceeds silently;
- loss of a documented path for v0.4 adopters to remain pinned or migrate;
- a CLI action that can manufacture human review, risk acceptance, or stage
  promotion.

The following do not block v0.5:

- #40 or #41 implementation;
- generic evidence envelope, SARIF, or vendor adapters;
- hosted services or dashboards;
- perfect text wording after the structured contract is correct;
- a full canonical rewrite when G1 says no-go.

## 14. Decisions required before public alpha

Record these as short ADRs:

1. **Product/package/CLI name.** The `AAP` acronym collides with other public
   meanings. Keep the full “OpenDevs Agentic Assurance Profile” name in public
   prose now; verify package and executable namespaces before publishing.
2. **Finding code shape and allocation.** Codes cannot contain severity or
   stage, cannot be recycled, and need a deprecation policy.
3. **JSON format and redaction.** Define schema versioning, stable fields,
   location semantics, sensitive-value policy, and deterministic order.
4. **Artifact-format identity.** Separate it from profile release and define
   current/N-1 support and migration chaining.
5. **Local engine/profile pin.** v0.5 starts with exact semantic matching;
   record how the packaged CLI detects and explains a mismatch.
6. **Git snapshot semantics.** Define worktree, index, commit, merge-base,
   missing-ref, shallow-clone, rename, symlink, and submodule behavior before
   releasing `--staged` or `--base/--head`.
7. **#43 event matrix.** Define `pull_request`, `pull_request_target`,
   `merge_group`, manual, schedule, and reusable-workflow caller behavior
   without introducing privileged execution of untrusted code.

## 15. v0.5 definition of done

v0.5.0 is done when:

- v0.4.0 remains an immutable, reproducible behavioral baseline;
- the Phase 0 evidence record exists and usability claims match its limits;
- every required check exposes applicability and completion;
- stable finding and JSON v1 contracts are documented and tested;
- common validation is reachable through one read-only command;
- workflow application logic after trusted checkout lives in tested modules,
  while the minimal pre-checkout trust bootstrap remains explicit;
- human authority cannot be fabricated by any CLI command;
- legacy v0.4 behavior has a documented compatibility or migration path;
- the full canonical engine either passes G1 and parity, or is explicitly
  deferred without holding the useful foundation hostage;
- rc.1 passes the corpus, security boundaries, self-check, dogfood fixture, and
  both recorded adopter trials;
- deferred issues and known limitations are visible in the release record.

The intended result is not a larger AAP. It is a smaller public surface over a
deeper, better-bounded core, with honest evidence for whether it is actually
easier to adopt.
