## 1. Executive decision

v0.5 should reduce the surface that adopters and maintainers must understand
while preserving the depth and honesty of the assurance model.

It should **not** begin with a full engine rewrite. It should begin with a
mandatory, independently useful foundation:

1. freeze the v0.4.0 fixture corpus and seed a reviewed expected-outcome
   ledger;
2. measure v0.4.0 adoption rather than infer its usability from repository CI;
3. give findings stable identities and a versioned machine contract;
4. extract the trusted post-checkout workflow application from YAML;
5. make check applicability explicit so a skipped check cannot look green;
6. put a thin CLI facade over the existing engine.

Only then does the project take a go/no-go decision on a canonical Snapshot
engine. A no-go is a valid outcome: the foundation above is enough to make a
useful v0.5 release without freezing an untested domain abstraction.

If G1 authorizes the full canonical track and Phase 3 parity completes, the new
engine may replace the legacy implementation through the Phase 5 cutover.
Loaders and typed findings move first, snapshot rules second, and transition
rules last. The frozen fixtures are inputs, a reviewed expected-outcome ledger
is the parity oracle derived from normative and accepted compatibility
decisions, and the v0.4.0 executable is comparison evidence rather than truth.

The v0.3.2 to v0.4.0 repository diff was 34 files and approximately

the growth is deliberate fail-closed, resource-bound, compatibility, and

Three corrections are important:

1. The 428 tests are not generally coupled to private function structure.
   They intentionally invoke the CLI and assert exit status and output. They
   should supply behavioral fixtures and comparison evidence, not be
   discarded; their existing assertions do not automatically become the
   oracle. The concentrated structural coupling is in tests that extract
   workflow steps by name, indentation, shell, and inline body.
2. The current `self-check` validates the central profile repository's
   VERSION, schemas, templates, and related invariants. It is not an actual
   adopter self-application because this repository has no adoption
   declaration. v0.5 needs a distinct dogfood adopter fixture.
3. Specialized active profiles inheriting `core` is already normative in
   [PROFILE.md §5–§6](../PROFILE.md). The canonical model may therefore derive
   `effective_profiles` from `declared_profiles`; it is not inventing policy.

The v0.4.0 reusable workflow runs drift only for `pull_request`. Push and other
caller events expose no drift verdict. Push is documented as outside the drift
check's scope; applicability for other event classes remains unresolved. A green non-PR
workflow must not be interpreted as a completed drift evaluation. This known
limitation is tracked in
[#43](https://github.com/MosslandOpenDevs/agentic-assurance-profile/issues/43)
and needs adopter-facing documentation independent of the v0.5 implementation.

### 2.3 INFERRED: architecture pressure

The same concepts are normalized repeatedly across lite/split layout, default
paths, implicit `core`, active/archived behavior, snapshot validation, and
base/head transitions. A typed boundary is likely to reduce this duplication
and make structured diagnostics, migration, and isolated Git snapshots safer.

This inference is strong enough to justify the foundation work. It is not yet
strong enough to justify making every proposed canonical type a public or
permanent compatibility contract.

### 2.4 UNKNOWN: adopter usability

The repository does not yet establish:

- how long an unfamiliar adopter needs to reach an owner-reviewable draft;
- whether the minimal lite route feels minimal in practice;
- which terms or decisions cause the most confusion;
- whether users get trapped in agent fix/fail loops;
- how many warnings are false or non-actionable;
- whether the dominant pain is authoring, classification, review, CI setup, or
  engine diagnostics;
- whether more than one real adopter needs the same new abstraction.

These are measurements for Phase 0, not assumptions that the rewrite may use
as premises.

## 3. Product definition

Working product sentence:

> Repository-native assurance for AI-assisted software change: check whether a
> repository's declared promises, supporting controls and evidence, and
> recorded risks remain represented after change, while keeping the
> declaration's review and authority status explicit.

The durable differentiation is not that no neighboring tool can ever cover
the same territory. It is a small, deterministic semantic transition engine
that can consume existing specs, tests, scanners, review results, provenance,
and attestations without pretending those outputs prove more than they do.

For an adopter, the mechanical gate evaluates the profiles and stage declared
by that repository. A passing result means that the declared-policy gate
completed without a blocking finding; it does not certify that the declaration
is complete, that its profile classification is correct, or that its claims are
true. Reports and CI status identities must preserve this boundary. The
under-classification backstop in #41 remains evidence-gated rather than being
silently implied by `aap check`.

### 3.1 v0.5 goals

- One obvious read-only entry point for the common check.
- Stable finding identity for CI, agents, documentation, and future adapters.
- Every check exposes applicability separately from completion and outcome, so
  a check that did not run can never appear green.
- Foundation-level `CheckPlan`, `CheckEvaluation`, `RunReport`, and
  adoption-bundle identities over the current engine.
- When G1 authorizes canonical scope, an internal model that represents lite
  and split artifacts consistently while preserving every physical source and
  revision.
- An in-memory compatibility boundary for existing v0.4 adoption bundles, with
  no forced on-disk migration in v0.5.
- A reusable workflow whose YAML contains only the necessary trust bootstrap,
  checkout, dependency setup, and module entry points.
- Deterministic, offline, resource-bounded, fail-closed behavior.

v0.5 work is split into three independently judged tracks:

1. **Foundation — always required:** typed findings, JSON, explicit check
   state, workflow extraction, and `aap check`.
2. **Progressive UX — evidence-gated, independent of G1:** preview-first
   `init` and read-only `review` may wrap the Phase 1 engine when Phase 0 shows
   that authoring or review friction justifies them.
3. **Canonical engine — conditional on G1:** Snapshot normalization, rule
   migration, and snapshot-dependent `migrate`, `--staged`, and
   `--base/--head` behavior.

A Foundation-only v0.5 is valid, but it MUST NOT claim preview-first adoption
or the `init -> check -> review` product path unless the Progressive UX
definition of done is also complete.

### 3.2 Anti-goals

v0.5 is not:

- a new normative assurance ontology;
- a certification, safety claim, bug-free claim, or security scanner;
- a spec generator, task manager, coding-agent orchestrator, or session recorder;
- a code graph, taint engine, generic SAST platform, sandbox, or runtime agent
  authorization system;
- a hosted dashboard or a large vendor-specific adapter collection;
- an automatic judge of human intent, truth, profile classification, residual
  acceptance, or approval;
- a reason to expose a full internal Snapshot, restricted policy content, or
  raw sensitive values in CI JSON;
- a requirement to eliminate every line of workflow bootstrap code.

## 4. What v0.5 preserves and what it may replace

The v0.4 hardening must be classified before code moves.

| Preserve independent of engine | Candidate for structural absorption |
|---|---|
| Bounded YAML/JSON/prose reads and node/depth/token limits | Repeated lite/split projection branches |
| Duplicate-key, merge-key, cyclic, non-finite, and malformed-input rejection | Repeated missing/`None`/list state conventions |
| Repository-relative path, containment, symlink, and trust-root checks | Repeated declared/effective profile derivation |
| Exact pin, checked-out revision, clean-resource, and canonical-tag checks | Compatibility fingerprints scattered through active rules |
| Injection-safe logs, annotations, and JSON; controlled diagnostics | Renderer strings used as rule identity |
| Stable IDs, stage semantics, profile obligations, and base/head protections | Workflow reimplementations of loaders and transition preparation |
| Fail-closed behavior when an applicable check cannot complete | Raw dictionaries crossing parser, semantic, and transition boundaries |

The goal is not fewer defensive tests. The goal is to express each defense
once at the correct boundary and stop multiplying states that later rules must
handle.

## 5. Delivery sequence

### Phase 0 — observe and close false-green ambiguity

This phase starts from immutable v0.4.0 artifacts, even though development
branches start from current `main`. It separates three things that must not be
conflated:

- the **fixture corpus** is a fixed set of input repositories and documents;
- the **expected-outcome ledger** is the reviewed parity oracle;
- the **v0.4.0 executable** is a reproducible reference implementation and a
  source of comparison evidence, but is not automatically correct.

Each fixture is classified `PRESERVE`, `KNOWN_BUG`, `INTENDED_CHANGE`, or
`UNSPECIFIED_LEGACY`. The ledger records the expected structured outcome and
rationale before parity work begins. Each entry cites PROFILE.md or an accepted
compatibility or bug-fix decision as its authority; the ledger is not an
independent normative source.

Phase 0 publishes an accepted `ledger_revision` and
`fixture_manifest_digest`. The fixture bytes and expected entries identified by
an accepted revision are immutable. A correction, `KNOWN_BUG`,
`INTENDED_CHANGE`, `UNSPECIFIED_LEGACY` resolution, or parity-allowlist change
creates a successor revision; it never rewrites an accepted revision in place.
Each successor is approved as a separate decision change and records its
decision ID, authority and review class, normative or compatibility rationale,
affected fixture digests, old and new expected outcomes, and compatibility
impact.

Deliverables:

1. Record a fixture manifest and expected-outcome ledger covering
   active/archived, lite/split with default and explicit mappings,
   current/legacy inputs, stage outcomes, transition regressions, path and
   trust failures, drift routing, and central self-check.
2. Add a real dogfood adopter fixture; do not rename central self-check as
   self-adoption.
3. Run at least two end-to-end v0.4.0 adoption trials: one existing brownfield
   adopter and one repository not constructed as an engine fixture. At least
   one trial SHOULD be performed or observed by someone other than the primary
   engine implementer, and an external adopter is a strong SHOULD.
4. Record the usability metrics in §12 and classify each problem as content,
   classification, authoring, review, CI integration, diagnostic, or engine
   structure.
5. Resolve #43's event contract before the engine rewrite: every check reports
   applicability, completion, and outcome separately.

For every trial, record whether it is internal dogfood, an affiliated adopter,
or an external adopter, together with the operator and the person making the
G1 decision. If an external trial cannot be obtained by the cutoff recorded in
the G1 issue, external usability evidence remains `UNKNOWN`; that does not
block Foundation work.

For #43, a v0.4.1 patch is appropriate only if the fix can make the existing
event contract explicit without adding a privileged execution path or breaking
supported callers. If safe handling requires new event permissions, new
base/head semantics, or `pull_request_target`, it belongs in the v0.5 CI
adapter after a threat-model review. PR-controlled code MUST NOT execute in a
privileged `pull_request_target` context.

### Phase 1 / alpha.1 — contracts before internals

This work is required regardless of the canonical-engine decision.

1. Introduce immutable `Finding`, `SourceRef`, a `CheckPlan` over public
   semantic capability checks, `CheckEvaluation`, gate-decision, and
   `RunReport` objects behind the current rules.
2. Allocate stable finding and check-reason codes and publish a mapping from
   legacy conditions to one of those structured identities.
3. Add JSON diagnostic schema v1 and a human text renderer.
4. Put a thin `aap check` facade over v0.4 behavior while retaining
   `scripts/validate.py` as a compatibility wrapper for the v0.5 line.
5. Introduce `AdoptionBundleFormatId` at the existing loader boundary and
   classify an unversioned compatible adoption bundle as `LEGACY_V0_4` in
   memory without serializing a change. Preserve a context-bound
   `PRE_V0_4_BASE` adapter only for historical base snapshots already supported
   by v0.4 transition checks; it is never a valid current/head format.
6. Extract post-trusted-checkout pin reading, change normalization, and the
   base/head materializer into package modules with direct tests.
7. Keep the pre-checkout identity check and checkout-destination reservation
   in YAML. There is no trusted local module before the fixed source checkout.
8. Preserve isolated execution (`python -I`), explicit trusted import paths,
   hash-locked dependencies, and adopter-import-shadowing defenses.

Deliver Foundation work through independently mergeable checkpoints: contract
core; CI, project-root, and trusted workflow extraction; then distribution,
bundle, and provenance compatibility. Each checkpoint preserves v0.4 normative
and blocking-gate semantics except for independently accepted `KNOWN_BUG` or
`INTENDED_CHANGE` ledger entries, creates no new normative obligation, and may
not expand another checkpoint implicitly. Diagnostic additions, including
classification-review reporting, are ledgered before implementation.

Before the first checkpoint, the Foundation umbrella fixes an aggregate effort
cap and child-issue order; each child also fixes its own effort limit and
stop/re-scope condition. At every checkpoint boundary, record actual and
remaining effort and a reviewed `CONTINUE | RESCOPE` outcome before opening the
next checkpoint. Active work cannot raise either cap. `RESCOPE` requires a
reviewed amendment to this design and its definition of done; it cannot silently
drop check-state, provenance, or trust-boundary obligations. A public package or
final v0.5 contract still waits for every applicable release requirement, so
checkpointing is not permission to publish an incomplete safety boundary.

At the end of alpha.1, finding messages may improve, but code, rule identity,
gate effect, source, `EvaluationKind`, `GateCoverage`, `GatePlanIdentity`,
planned public `check_id`, `completion_required`, the three check-state axes,
`reason_code`, and public check dependency edges become the versioned parity
comparison contract. This alpha baseline is not the final public v0.5
compatibility freeze: separately reviewed pre-release revisions remain possible
until the rc freeze in Phase 5. Every such revision bumps its pre-release
schema or catalog revision, updates the legacy mapping and migration note, and
reruns the affected corpus.

A stable public `check_id` names a semantic capability, not an implementation
step. Parser, loader, materializer, repository-I/O, and scheduler operations may
use internal `task_id` values and an internal task graph for diagnostics, but
those identifiers and edges are not finding-protocol or parity contracts.
Public `blocked_by` relationships refer only to planned public capability
checks. The versioned gate-plan manifest may change its public capability graph
through the reviewed plan-version process; an internal refactor alone does not
require a public graph change.

### Gate G1 — canonical engine go/no-go

The recorded outcome is `GO(scope)`, `DEFER`, or `NO_GO`. A scoped GO names
the exact rule families and snapshot capabilities it authorizes. A limited GO
does not authorize default-engine replacement. Default cutover requires ledger
parity for every active runtime behavior and a separate recorded cutover
decision; expanding scope requires another reviewed G1 record.

Before writing the spike, create a reviewed G1 pre-registration that defines
and freezes its acceptance cases, evidence thresholds, reviewer relationship,
and effort cap. This pre-registration is not the final G1 decision record that
freezes this design. Relaxing a frozen condition requires discarding the spike
and starting a new reviewed pre-registration; the active spike cannot raise its
own budget. The spike is disposable, does not enter the production runtime, and
is stopped as `DEFER` if the cap is exceeded. After the bounded evidence is
reviewed, the final G1 record links the pre-registration and records
`GO(scope)`, `DEFER`, or `NO_GO`. Exact calendar or effort limits belong there
rather than in this long-lived design.

`Verified structural duplication` is a reproducible measurement, not a label
assigned after seeing a favorable spike. Its pre-registration names the exact
semantic transformation, at least two independent production consumers across
at least two rule or workflow families that reconstruct it, source locations
and fixtures that demonstrate equivalent meaning, baseline change
amplification, a preselected collapse threshold, and the approving reviewer,
who must be distinct from the spike implementer. Without
that separation the evidence remains `INFERRED` and cannot use this independent
GO route. A necessary snapshot-dependent feature similarly names the accepted
requirement and shows why documentation or the Phase 1 facade cannot provide
it. If either is the independent basis for `GO(scope)`, the final record keeps
adopter usability `UNKNOWN` where evidence is absent and limits scope to the
measured seam, named rule families, and required dependencies.

The structural-duplication route is satisfied only when the spike meets the
pre-registered collapse threshold, replaces the named reconstructions behind
one bounded seam, and produces no unexplained ledger difference; otherwise that
scope records `DEFER` or `NO_GO`. A necessary-feature route must cite an
independently accepted normative, compatibility, or user requirement and prove
its pre-registered spike case under the same ledger rule; naming a desired
Snapshot capability without that result is not GO evidence.

Proceed to Phase 2 only when all of the following are true:

- **Parity ready:** every supported fixture has a reviewed ledger entry;
  unexplained differences from that expected outcome are zero.
- **Evidence ready:** the trials have complete notes and disclose operator,
  decision-maker, and adopter relationship. At least one repeated problem
  cannot be solved only by documentation or a CLI facade, **or** verified
  structural duplication or a necessary snapshot-dependent feature provides
  an independent reason for a limited engine move.
- **Spike ready:** one bounded vertical slice handles a normal core-lite input,
  a specialized split input with explicit mappings, malformed or partial input
  that produces no canonical snapshot and marks dependent checks `NOT_RUN`, and
  a base/head register transition while preserving `SourceRef` and revision.
  It also proves `[service]` and `[core, service]` have the same effective
  obligations without erasing their different declarations.
- **Scope clean:** no new normative obligation is mixed into the engine move.
- **Fallback viable:** the project can still release the Phase 1 foundation if
  the prototype is stopped.

An external pilot is strong evidence, not an absolute prerequisite. If all
trials are internal and usability observation is the only GO evidence, the
default decision is `DEFER` or `NO_GO`, and any weaker conclusion is labeled
`INFERRED`. Verified structural duplication or a necessary snapshot feature
may independently justify a deliberately limited GO.

If any condition fails, stop the full rewrite. Ship the boundary, finding, CI,
and CLI work as v0.5.0, collect more adoption evidence, and revisit the engine
in a later minor. This is not a failed v0.5.

### Phase 2 / alpha.2 — canonical model behind the current engine

Introduce the model as an internal seam, not a new public file format:

```text
RawDocument
  + evaluation context
  -> bounded format/context probe
  -> StrictParsedDocument
     | quarantined LegacyBaseParsedDocument (PRE_V0_4_BASE only)
  -> version-aware AdaptedDocument
  -> PartialSnapshot
  -> CanonicalSnapshot
  -> snapshot and transition rules
  -> CheckEvaluation(state, findings)
  -> GateDecision and RunReport
  -> text / JSON / GitHub adapters
```

The types and sequence below are design hypotheses, not spike acceptance
criteria. A simpler or materially different internal model that satisfies the
pre-registered behavioral, provenance, and trust-boundary cases is a successful
result. The final G1 record describes the difference, and an accepted ADR fixes
the production model before implementation proceeds; the spike is not required
to confirm this document's proposed type shape.

Every stage returns an outcome with findings and completion state. Malformed or
unsafe input need not produce a CanonicalSnapshot; it must produce a controlled
failure and mark dependent checks `NOT_RUN`, never silently pass them.

The legacy-base parser is authorized only by a bounded probe plus historical
base context already accepted by the v0.4 transition contract. Its permissive
semantics never parse a current/head document and do not leak into the strict
parser. Every accepted legacy class appears explicitly in the ledger.

The minimum domain model includes:

- `AdoptionMode = Active | Archived`;
- `ArtifactLayout = LITE | SPLIT`;
- `PolicyPathBinding(binding_id, roles, raw_declared_path,
  normalized_lexical_path, resolved_target_or_failure, expected_kind,
  path_origin, content_state, impact_eligible, declaration_source_ref)`, where
  roles cover the adoption declaration, root guides, AAP artifact content,
  specification-workflow root, human-review record, security/public roots, and
  local extension paths;
  `expected_kind = FILE | DIRECTORY | FILE_OR_DIRECTORY`,
  `path_origin = FIXED | DEFAULTED | EXPLICIT`, and
  `content_state = ACTIVE_CONTENT | DORMANT_POLICY | NOT_CONTENT`;
- `ArtifactContentSource = InlineNode(SourceRef) |
  PathBindingRef(binding_id)` for each applicable artifact role;
- a `PolicyPathBindingSet` plus an `ArtifactSet` of selected content sources,
  with derived `has_explicit_path_mappings`; mapping is not a third layout, and
  a non-applicable artifact does not receive a fabricated content source;
- both `declared_profiles` and spec-derived `effective_profiles`;
- each partial/canonical snapshot's adoption stage, profile pin,
  adoption-bundle identity, revision identity, and register graph, with base
  and head represented independently for transitions;
- `SourceRef(uri, pointer, line, column, revision, source_role, trust_domain)`
  where known; source role and trust domain preserve whether bytes came from
  the adopter tree, pinned profile, or generated temporary materialization and
  how that boundary was acquired;
- `ValueOrigin = EXPLICIT | DEFAULTED | INHERITED | SYNTHESIZED`, separate
  from an ordered adaptation/transformation history and source-provenance
  chain, so a transformed explicit value remains explicit without hiding that
  it was adapted.

The separation between policy path binding and selected content source is
required for lite documents. An inline system may be the active content while
an explicit mapped path is dormant. That dormant mapping still remains policy:
containment, symlink retargeting, base/head relocation, and impact-eligibility
checks continue to evaluate it even though its bytes do not supply the current
content. Raw spelling, normalized lexical identity, and resolved target are
kept separately; the engine never normalizes away `..` or symlink semantics
before resolving the raw path against its trust root.

Unknown line or column is `null`; the engine never invents a location. A
normalized value retains its physical source so path relocation, symlink,
review-boundary, and base/head checks remain possible.

`AdaptedDocument` is an in-memory compatibility view used by `check`.
`migrate` is the separate operation that serializes a reviewed representation.
`EXPLICIT` may be assigned only when a direct lexical node and `SourceRef`
exist. Adaptation alone never upgrades a defaulted, inherited, or synthesized
value to explicit, and it preserves the prior origin and source chain.
Origin and source trust record data provenance and acquisition boundaries;
neither is evidence of human authority, review, or ownership.

### Phase 3 / beta.1 — move rules under parity

Within the exact G1-authorized scope, move the named rule families and their
required dependencies in the following order, and stop at the scope boundary.
A full-scope GO uses the complete sequence:

1. parsing, schema, and bounded repository reads;
2. profile composition, required artifacts, and layout normalization;
3. snapshot semantics and stage rules;
4. component routing and change facts;
5. adoption, path, and register transition regressions.

For each family, run the v0.4 reference executable and the new engine on the
same corpus inputs. Compare both with the reviewed expected-outcome ledger,
not with each other as an implicit source of truth. Compare structured semantic
results, not full prose formatting. The legacy implementation receives no new
feature rules.

Parity CI for an implementation change MUST use the ledger revision accepted on
its base branch or another independently accepted decision commit. An
implementation change MUST NOT establish its own expected outcome or parity
allowlist in the same change. A required ledger change is accepted first as a
successor revision and only then becomes an implementation baseline.

Only `KNOWN_BUG` or `INTENDED_CHANGE` differences with a reviewed migration
note may enter the parity allowlist. An entry states the reference-executable
outcome, ledger outcome, the normative rule or accepted operational or
compatibility rationale, affected fixtures, and compatibility impact.
`UNSPECIFIED_LEGACY` must be resolved before the affected behavior can claim
parity.

### Phase 4 / beta.2 — progressive UX

After the diagnostic contract is trustworthy, Phase 0 evidence may authorize
the Progressive UX track independently of G1. `init` and read-only `review`
may wrap the Phase 1 engine; they do not require a canonical Snapshot. If the
track ships, the normal route is `init -> check -> review`.

Before implementation, the UX authorization record selects its primary
metrics, minimum acceptable improvement, and non-regression guards. The team
does not choose favorable metrics after seeing the result.

Snapshot-dependent `migrate`, isolated `--staged`, and `--base/--head`
features remain conditional on G1 and their own threat-model and semantics
ADRs. `explain` may ship with the Foundation finding catalog.

This phase runs the same adoption trials again and compares the measured
results with Phase 0. It does not claim better usability merely because the
command is shorter.

### Phase 5 / rc.1 and final

## 6. Architecture boundaries

The always-required Foundation stabilizes the right side of the engine without
pretending the current rule implementation already has canonical boundaries:

```text
current-engine adapter -> CheckEvaluation -> gate -> RunReport -> renderers
             ^
             |
  provenance and trusted CI event/materialization facts
```

The target dependency direction for each G1-authorized canonical scope is:

```text
bounded document and repository adapters
              |
              v
     partial/canonical snapshots
              |
              v
   snapshot rules -> transition rules
              |
              v
 CheckEvaluation { check state, findings }
              |
              v
 operational gate policy -> RunReport
              |
              v
 text / JSON / GitHub / future SARIF renderers
```

A rule family migrated into the canonical scope does not read arbitrary files
or raw YAML. Foundation-only v0.5 does not claim that the untouched legacy
rules already satisfy this boundary. Renderers do not decide policy.
Gate policy evaluates severity and blocking effect from explicit rule context,
but it does not change finding identity.

Rule context includes declared/effective profiles, current and base stage,
visibility with its provenance, policy acknowledgment, invocation scope, and
trusted event facts where applicable. Operational configuration is monotone
only after that context is evaluated:

```text
profile_floor = derive_from(PROFILE.md, adoption_stage, rule_context)
compatibility_floor = accepted_compatibility_effect(rule_context)
normative_minimum = stronger_of(profile_floor, compatibility_floor)
effective_gate_effect = stronger_of(normative_minimum, configured_effect)
```

PROFILE.md and the applicable adoption stage establish the normative floor.
An accepted compatibility contract may preserve an additional stricter effect,
but may neither subtract from nor override that floor. Configuration MAY
strengthen the resulting minimum and MUST NOT weaken it. Completion-required
check incompletion bypasses gate-effect ordering and is resolved by the
`OverallOutcome` precedence in §7.1.
The public `aap check` exposes no gate-narrowing mode. A fixed internal
workflow invocation may evaluate `GateCoverage = PARTIAL`, but its report must
expose that coverage and cannot call the result a complete mechanical gate.
No tool-generated report by itself claims repository conformance.

A practical split is:

| Boundary | Track | Responsibility |
|---|---|---|
| `finding` | Foundation | immutable semantic Finding, bounded facts, SourceRef, and deterministic ordering |
| `gate` | Foundation | check state, evaluated severity, normative minimum, gate effect, and overall outcome |
| `renderers` | Foundation | bounded/redacted text, JSON, and GitHub projections; no policy decisions |
| `repo` | Foundation/shared | containment, symlink, Git object, revision, and trust-boundary reads used by extracted CI and any canonical scope |
| `provenance` | Foundation | package/profile/pin matching and verified source identity |
| `ci` | Foundation | caller event facts and isolated base/head materialization |
| `commands` | Foundation plus authorized extensions | self-check, check, review preparation, migration, compatibility entry points |
| `policy_io` | G1-conditional | bounded YAML/JSON/prose parsing, schema registry, source marks |
| `snapshot` | G1-conditional | declared/effective profiles, defaults, lite/split projection, domain values |
| `rules` | G1-conditional | snapshot, semantic, and stage rules |
| `transition` | G1-conditional | adoption/path/register regressions and base/head comparison |

These names are illustrative. For an authorized canonical scope, the
architectural test is that its snapshot and transition rules cannot reach back
into raw YAML or the ambient filesystem, not that the repository contains a
particular number of files.

PROFILE.md stays normative. A mechanical rule catalog may map rule ID,
applicability, required artifact, transition kind, and finding code, but it
must not silently replace or generate the normative prose.

## 7. Finding, check, and gate contract

A finding code identifies a semantic condition. It does not encode severity,
stage, renderer, or the actor believed able to fix it.

The stable JSON v1 budget is deliberately small:

- `finding`: stable `code`, owning `check_id`, a typed `reference`, bounded
  semantic `facts`, and zero or more source `locations`; `reference` is one of
  a normative `rule_id`, an accepted compatibility-decision ID, or a versioned
  operational-catalog ID;
- `evaluation`: evaluated `severity` and `gate_effect`;
- optional `resolution`: `authority_required` and
  `automation = PROPOSE_ONLY` where the rule can state them honestly.

If `resolution` is absent, a caller MUST NOT infer permission to modify policy
or apply a fix. `auto_apply`, renderer prose, and redaction metadata are not
part of the stable core. Category, title, and default severity may live in the
versioned rule catalog. Each finding code defines the allowed fact keys and
value types in that catalog; “bounded facts” is not an open-ended extension
map. A human `message` may be included as an optional display projection, but
its wording is not a stable machine contract. Redaction, bounding, escaping,
and deterministic ordering are serialization invariants, not public policy
fields.

Resource, dependency, import, I/O, Git-snapshot, or unsupported-event failures
that prevent an evaluation normally use the affected check's stable
`reason_code` rather than inventing a normative finding. The compatibility
mapping therefore maps each legacy condition to either a finding code or a
check reason code.

Minimum run data:

- tool/build/package provenance, bundled profile release and commit, and
  diagnostic-format versions;
- `EvaluationKind`, `GateCoverage`, and `GatePlanIdentity`, plus a bounded
  `selection_id` or `selection_digest` when coverage is partial;
- each requested current/base/head input as an `InputReport` containing
  `IdentityState = KNOWN(value) | UNDETERMINED(reason_code)`, revision, profile
  pin, adoption-bundle format, declared profiles, spec-derived effective
  profiles, adoption stage, and
  `ProfileClassificationReview = UNCONFIRMED | REVIEW_RECORDED |
  UNDETERMINED(reason_code)`, with the relevant declaration and review
  `SourceRef` values when known; the run summary repeats the primary
  current/head declaration state rather than conflating it with the base;
- every planned public capability `check_id` and its public dependency edges;
- every planned check's stable `check_id`, `completion_required` flag,
  applicability, completion, outcome, state-dependent `reason_code`, and zero
  or more public `blocked_by` check IDs;
- findings, summary, `OverallOutcome`, and exit code.

The public semantic capability check
`adoption.profile-classification-review` evaluates the primary current/head
declaration. It derives:

- `REVIEW_RECORDED` only when a safely loaded `HUMAN_REVIEWED` or `CONFORMANT`
  declaration has a qualifying durable review binding that mechanically passes
  its stage rules;
- `UNCONFIRMED` when the declaration is safely loaded but that condition is not
  met, including at `DRAFT`; and
- `UNDETERMINED(reason_code)` when acquisition or evaluation cannot classify
  the state safely.

`REVIEW_RECORDED` records mechanical review provenance; it does not mean the
tool independently proved human authority. For `UNCONFIRMED`, the completed
capability check owns a stable non-blocking unresolved-decision finding. For
`UNDETERMINED`, that check has no finding and instead reports its stable
non-completion reason under §7.1. The capability is completion-required for
adopter evaluation kinds, while its completed finding has a maximum gate effect
of `WARN`; an undetermined state therefore makes the run `INDETERMINATE` rather
than silently green. Base-input states remain distinct transition facts and do
not masquerade as the primary declaration's review state. Text and CI output
MUST NOT reduce either `UNCONFIRMED` or `UNDETERMINED` to an unqualified
conformance statement. This capability reports review provenance only; it does
not scan code triggers or implement #41. Its diagnostic addition is
pre-registered as an `INTENDED_CHANGE` ledger entry before implementation.

The public JSON report MUST NOT expose the full internal CanonicalSnapshot.
That would prematurely freeze the private model and could leak restricted or
embargoed content through CI artifacts. JSON is an AAP diagnostic projection;
SARIF, if added later, is another intentionally lossy projection.

Text may use readable sections such as `Required`, `Observed`, and `Next step`,
but text layout and wording are not the machine API. A diagnostic may omit a
section that does not fit. A next step may prepare or request a decision; it
must not grant authority.

### 7.1 Gate plan, coverage, and check state

Run-level state is:

```text
EvaluationKind = ADOPTER_SNAPSHOT | ADOPTER_TRANSITION | CENTRAL_SELF_CHECK
GateCoverage   = COMPLETE | PARTIAL
OverallOutcome = PASS | FAIL | INDETERMINATE

GatePlanIdentity =
  (plan_id, plan_version, plan_digest,
   engine_build_identity,
   bundled_profile_release, bundled_profile_commit,
   source = TRUSTED_ENGINE_PROFILE_PLAN)
```

`EvaluationKind` identifies the mechanical gate being evaluated. A snapshot
evaluation cannot claim completion of a transition gate. A dogfood adopter uses
the applicable adopter evaluation kind and records dogfood as fixture or run
provenance; dogfood is not a separate gate kind.

`GatePlanIdentity` always identifies the trusted canonical target plan for the
recorded `EvaluationKind`. For `GateCoverage = PARTIAL`, the planned public
check IDs and edges identify the actual subset. A stable partial projection uses
its separate bounded `selection_id` or `selection_digest`; it does not overload
the canonical `plan_digest`.

`GateCoverage = COMPLETE` is valid only when:

- the plan is obtained from verified engine or package and bundled-profile
  provenance;
- `plan_id` and `plan_version` identify the canonical plan for the recorded
  `EvaluationKind`, and `plan_digest` matches its trusted manifest;
- every public check and dependency edge named by that canonical plan is present
  in the instantiated `CheckPlan`; and
- adopter-controlled input has not removed or rewritten a plan entry, dependency
  edge, or the trusted plan builder's derived `completion_required` value.

Adopter facts are inputs to the trusted deterministic derivation of
applicability and `completion_required`; they never remove canonical plan
entries or override the derived result. Any intentionally selected subset is
`GateCoverage = PARTIAL`. Coverage describes the plan, not whether execution
completed: a complete plan may still finish `FAIL` or `INDETERMINATE`.

A partial report describes only its declared subset and cannot be rendered as
“the gate passed.” Repository conformance remains the adopter's declared stage
plus the applicable human authority and substantive obligations.

Every check then reports three orthogonal axes:

```text
Applicability = APPLICABLE | NOT_APPLICABLE | UNDETERMINED
Completion    = COMPLETED | NOT_RUN | FAILED_TO_COMPLETE
Outcome       = PASS | FAIL | NONE
```

For an adopter evaluation, `GateCoverage = COMPLETE` plus
`OverallOutcome = PASS` means only that the canonical plan's checks, as
evaluated for applicability under the repository's declared profiles,
spec-derived effective profiles, and stage, completed without a blocking
finding. It does not establish that the declared profile set covers every
trigger that applies. Even `REVIEW_RECORDED` may be under-classified; detecting
that condition remains #41's separately measured, warn-first discovery scope.

`completion_required` means that safe completion is required within the trusted
gate plan. `CheckPlan` is closed under completion-required dependencies: every
transitive prerequisite of a completion-required check is itself
completion-required.

For the invocation's trusted rule context, every check that can produce an
effective `gate_effect = BLOCK` is completion-required. If plan construction
cannot prove that a check's maximum effective gate effect is no stronger than
`WARN`, it MUST mark that check completion-required. Operational configuration
that strengthens a possible finding to `BLOCK` therefore also makes the owning
check completion-required before execution. Once the trusted plan builder has
derived the flag, no caller override, execution failure, or missing dependency
can lower it. A completion-optional check has a maximum effective gate effect of
`WARN`.

Valid combinations are constrained:

- `NOT_APPLICABLE` requires `NOT_RUN + NONE` and a stable `reason_code`;
- `APPLICABLE + COMPLETED` requires `PASS` or `FAIL`;
- `APPLICABLE + NOT_RUN/FAILED_TO_COMPLETE` requires `NONE`;
- `UNDETERMINED` requires `NOT_RUN` or `FAILED_TO_COMPLETE`, plus `NONE`;
- every not-applicable or non-completed check carries a stable `reason_code`
  and may identify zero or more prerequisite checks in `blocked_by`.

Every finding belongs to one planned `check_id`. For an
`APPLICABLE + COMPLETED` check, `Outcome = FAIL` if and only if at least one
attached evaluated finding has effective `gate_effect = BLOCK`; otherwise its
outcome is `PASS`. This prevents a blocking finding from coexisting with a
green owning check. Evaluated findings are allowed only on
`APPLICABLE + COMPLETED` checks. Not-applicable and non-completed checks use
their reason code and an optional bounded display detail, so a blocking finding
cannot be hidden on a check whose outcome is `NONE`.

Overall precedence distinguishes a safely evaluated policy failure from an
independent inability to evaluate:

1. A completion-required `NOT_RUN` with
   `reason_code = PREREQUISITE_FAILED` does not by itself make the run
   indeterminate only when `blocked_by` is non-empty,
   references existing checks, is acyclic, and every chain ends in a completed
   blocking `FAIL`. A schema failure may therefore return exit `1` while
   dependent semantic checks truthfully report `NOT_RUN`.
2. Any other non-completed completion-required check — including
   `APPLICABLE + NOT_RUN/FAILED_TO_COMPLETE` or `UNDETERMINED` — makes the
   overall run `INDETERMINATE`, even when separate policy findings also exist.
3. Otherwise, any completed blocking failure in the declared plan makes the
   overall run `FAIL`; if none exists, the overall run is `PASS`.
   Completion-optional check incompletion remains visible in the summary
   without making the run indeterminate. By the plan invariant above, such a
   check cannot produce a blocking finding.

Warnings are evaluated findings, not a third check outcome. A completed check
may pass with warnings; a check that did not complete cannot pass.

### 7.2 Exit status

The new CLI uses outcome classes, not guessed remediation actors:

| Exit | Meaning |
|---:|---|
| `0` | Every completion-required check is `COMPLETED` or validly `NOT_APPLICABLE`, and no completed check produced a blocking finding; warnings and completion-optional incompletion may exist. For `GateCoverage = COMPLETE`, the mechanical gate for the recorded `EvaluationKind` passed. |
| `1` | Every completion-required check is `COMPLETED`, validly `NOT_APPLICABLE`, or validly blocked by `PREREQUISITE_FAILED`, and at least one completed check produced a blocking finding. For `GateCoverage = COMPLETE`, that mechanical gate is blocked. |
| `2` | CLI usage or option-combination error. |
| `3` | A completion-required evaluation in the declared plan could not complete safely: dependency, I/O, Git snapshot, unsupported applicable event, or internal failure. |
| `128 + signal` | Conventional signal termination. |

Exit `3` means indeterminate, not “human approval required.” Except for the
fully explained `PREREQUISITE_FAILED` case in §7.1, a run containing both
policy findings and an incomplete completion-required check returns `3`,
because it cannot produce a complete mechanical gate result. Malformed policy
that was safely acquired and rejected by the bounded parser — including invalid
UTF-8 or data model, YAML syntax, duplicate keys, or byte/node/depth budgets —
is a completed parse-check `FAIL` and returns `1`, as do deterministic missing
or invalid adopter artifacts that a rule can classify. Acquisition, dependency,
internal, deadline, or Git-snapshot failure that prevents the check from
classifying the adopter input returns `3`. Any deliberate departure from the
v0.4 boundary is an `INTENDED_CHANGE` ledger entry.

Whether an agent may propose a mechanical change or must request human-owner
judgment belongs to an optional finding `resolution`. It is never inferred
from the process exit code, and v1 never grants automatic application.

### 7.3 Output channels

- `stdout` contains exactly the selected result: human text or one UTF-8 JSON
  document.
- `stderr` contains usage, bootstrap failure, and opted-in diagnostics.
- JSON never contains ANSI color, progress logs, or GitHub workflow commands.
- GitHub annotations are produced by an explicit CI adapter, not automatically
  because `GITHUB_ACTIONS` exists.
- A CI adapter publishes a status identity derived from `EvaluationKind` and
  `GateCoverage`. A `GateCoverage = PARTIAL` report MUST NOT publish, reuse, or
  satisfy the status identity reserved for the corresponding complete gate.
  When an AAP status is configured as required, repository rulesets and branch
  protection use only the stable complete AAP status identity for the intended
  `EvaluationKind`; this does not constrain unrelated required checks. A partial
  status is distinct and non-substitutable regardless of process exit or check
  conclusion.
  Illustrative names are `assurance / complete:<evaluation-kind>` and
  `assurance / partial:<selection-id>`; exact bounded spelling belongs to the
  gate-plan/CI-status ADR, and event mapping belongs to the #43 event ADR.
- `--quiet` applies only to text and suppresses successful/non-blocking output.
- text honors `NO_COLOR` and `--color auto|always|never`.

The bootstrap must determine output mode using the standard library before
optional dependency imports, so a missing dependency cannot corrupt requested
JSON with an unstructured line.

## 8. CLI and human-authority UX

### 8.1 `aap check`

Default behavior is read-only, deterministic, network-free, and repository
wide. It discovers the adoption declaration and pinned profile context instead
of requiring users to pass schema and checkout paths.

The project root defaults to the top level of the current Git worktree. Outside
a Git worktree, or when the caller needs another root, `--project-root <path>`
is required. The root is resolved and fixed before any policy read; every
default and custom adoption path is interpreted relative to it.

Discovery is deterministic: only `.agentic-assurance/adoption.yaml` is used by
default. A missing default is a controlled error. A custom declaration requires
`--adoption <path>`. It must be repository-relative and pass the same lexical
containment, resolved-target, symlink-retarget, and trusted-root checks as other
policy paths. For v0.4 compatibility, harmless legacy spellings such as one
leading `./` or contained `..` are normalized only after raw-path resolution;
the raw spelling and canonical lexical identity are both retained. New output
uses the canonical spelling, and any future tightening is an `INTENDED_CHANGE`
ledger entry. The CLI never recursively searches for a declaration or chooses
the nearest one based on the current directory.

Initial options:

- `--format text|json`;
- `--quiet` and `--color` for text;
- `--project-root <path>` and `--adoption <repository-relative-path>` for
  explicit roots;
- `--repo-visibility public|private|internal`, preserving v0.4's conservative
  behavior: omitted visibility is unknown and is evaluated as public where
  disclosure rules require a choice;
- `--base <ref> --head <ref>` only after Git snapshot isolation exists;
- `--staged` only when base=`HEAD` and head=`index` reads every policy artifact
  from the index, never the mixed worktree;
- `aap explain <finding-code>` for rule help.

The CLI does not implicitly fetch. Missing refs, shallow history that prevents
the required comparison, a conflicted index, or an unsafe snapshot is exit `3`.
The report records whether visibility came from a caller assertion or a trusted
CI event context; the local CLI does not infer it through a network call.

Do not add `--profile <path>` as a gate-narrowing option. `profile` already
means `core/service/...`, and partial validation could suppress an obligation.
A future `--focus` may filter display while still evaluating the complete gate;
the public v0.5 command has no option that narrows the gate. Fixed internal
partial invocations follow §7.1 and cannot report repository conformance.

### 8.2 `aap init`

This command ships only if the Progressive UX track is authorized and meets
its own definition of done; it may wrap either the Phase 1 or canonical engine.
`init` lowers authoring cost without claiming to know the adopter's truth.

- Preview is the default; filesystem changes require `--apply`.
- It scans for profile-trigger evidence and proposes candidate classification
  with `file:line` support, biasing ambiguous cases toward owner review.
- It may create a DRAFT with `UNKNOWN` conclusions and unresolved decision
  prompts.
- It never auto-creates `VERIFIED`, accepts a residual, closes a defeater,
  promotes a stage, or asserts a human decision.
- Previewed classification remains a proposal until the repository's durable
  owner-review process confirms it. Once a generated declaration is selected
  by `aap check` or committed as the adoption declaration, its declared and
  effective profiles are evaluated as self-binding policy even while owner
  confirmation remains an unresolved review decision.
- `--apply`, committing the proposal, or selecting it for `aap check` never
  upgrades `ProfileClassificationReview`. Only a qualifying durable review
  record may produce `REVIEW_RECORDED`; tooling reports that record but does not
  convert its own invocation into human authority.
- The tool records what it defaulted, inherited, adapted, or synthesized.

[#41](https://github.com/MosslandOpenDevs/agentic-assurance-profile/issues/41)
may later improve this proposal layer, but a heuristic never becomes the
canonical effective profile. The actual declaration is the mechanical policy
input whenever it is checked; human confirmation supplies classification
authority. Code signals remain advisory evidence and possible escalation.

### 8.3 `aap review`

This command has the same Progressive UX release condition as `init`.
`review` is read-only by default. It produces an owner review packet containing
unresolved decisions, affected stable IDs, cited evidence, transition findings,
and concrete questions.

`aap review prepare --write-draft` may write a clearly non-authoritative draft
when explicitly requested. There is no `aap review --approve` command. A local
TTY, username, confirmation prompt, or agent invocation is not evidence of
human authority. The owner's actual review/approval in the repository's durable
process creates the authority; tooling may later verify and record that event,
but not manufacture it.

### 8.4 `aap migrate`

This command is Snapshot-dependent and remains conditional on G1.
Migration previews by default and changes files only with `--apply`.

It preserves stable IDs and represented semantics. It may make a mechanical
format change, but it never invents intent, evidence, classification, review,
status, defeater disposition, residual acceptance, or risk truth. Any such
missing decision is emitted as a review item.

## 9. Versioning and compatibility

Version these contracts separately:

1. **profile release** — normative profile and bundled implementation release;
2. **adoption bundle format** — the adoption declaration, layout/binding
   composition, and the schema versions of its AAP-owned structured documents;
3. **finding protocol** — JSON report and stable diagnostic semantics;
4. **evidence envelope** — a future integration contract, not part of v0.5
   merely because MAPPINGS §5 exists.

The adoption bundle must stop inferring all compatibility from
`upstream.version`. Existing lite and register documents already carry their
own `version: 1`; the bundle identity composes those schema versions with the
currently unversioned adoption declaration and its layout/binding semantics.
Arbitrary extension paths, prose, and directories do not acquire invented
document versions.

v0.5 introduces an internal adoption-bundle identity, but does not force
existing repositories to rewrite documents. The exact field placement and
first explicit bundle version are an ADR.

Compatibility rules:

- immutable v0.4.0 adopters continue using the pinned v0.4.0 workflow/validator;
- `scripts/validate.py` remains a v0.5 compatibility wrapper;
- after an explicit owner-reviewed profile-pin upgrade to v0.5, an otherwise
  unchanged unversioned v0.4 adoption bundle loads in memory as
  `LEGACY_V0_4` without changing adopter files;
- bundle adaptation never overrides profile-pin matching: a repository still
  pinned to profile v0.4 continues to use the v0.4 validator;
- `PRE_V0_4_BASE` is accepted only as a context-bound historical base for the
  transition compatibility that the ledger preserves. It cannot be loaded as
  the current/head adoption, emitted by `init`, or written by `migrate`;
- new `init`, if the Progressive UX track ships, writes the first explicit
  adoption-bundle identity;
- `migrate`, if G1 authorizes it, previews adding an explicit identity and
  writes only with `--apply`;
- explicit identity becomes mandatory only when a later incompatible format
  cannot be identified safely from the pinned legacy contract;
- v0.5 text wording may change, but a published mapping explains every legacy
  condition's finding code or check reason code;
- current/N-1 support and chained-migration policy starts after two explicit
  bundle formats exist; it does not evict the unversioned v0.4 baseline in
  v0.5;
- a v0.5 local mechanical check requires an engine/profile release match rather
  than silently evaluating a pinned profile with “latest” semantics;
- the reusable CI workflow continues executing the exact full-SHA trusted
  checkout, which is the authoritative way to avoid an installed-tool mismatch;
- migration never silently upgrades a repository's human-approved meaning.

## 10. Test strategy

### 10.1 Keep the 428-test behavior asset

Classify existing tests instead of replacing them wholesale:

- **behavioral/characterization:** supported input, exit, semantic finding, and
  compatibility behavior; survives the rewrite;
- **trust/integration:** Git, path, symlink, resource, injection, and subprocess
  boundaries; remains integration-heavy;
- **workflow structural:** exact YAML step names, indentation, shell, and inline
  body; intentionally rewritten as module tests plus a small wiring contract;
- **renderer compatibility:** a small legacy text subset and the new stable
  structured contract.

Each corpus fixture first receives its `PRESERVE`, `KNOWN_BUG`,
`INTENDED_CHANGE`, or `UNSPECIFIED_LEGACY` ledger entry and stable-code
mapping. New parity asserts check state, finding code, evaluated severity, gate
effect, relevant source, `EvaluationKind`, `GateCoverage`, `GatePlanIdentity`,
planned public check IDs and `completion_required` flags, all three check-state
axes, reason code, public `blocked_by`, and exit class against the accepted
ledger revision. Parity also asserts that the public dependency edges in the
instantiated plan match the trusted manifest. It does not require byte-for-byte
preservation of every human sentence.

### 10.2 Add focused unit and property tests

High-value properties are:

- normalization is idempotent;
- lite and equivalent split inputs produce equal semantic snapshots while
  retaining different SourceRefs;
- `[service]` and `[core, service]` have equal effective obligations while
  preserving different declarations;
- in-memory adaptation and serialized migration preserve stable IDs, semantic
  values, prior value origin, and source-provenance chains;
- `EXPLICIT` always has a direct lexical node and `SourceRef`; adaptation alone
  never makes a defaulted, inherited, or synthesized value explicit;
- provenance origin is never interpreted as human authority or ownership;
- policy bindings preserve raw spelling, normalized lexical identity, and
  resolved target separately; dormant lite mappings remain trust- and
  transition-checked but never become active content accidentally;
- `PRE_V0_4_BASE` parses only in authorized historical-base context and is
  rejected as current/head input;
- unchanged base/head snapshots create no transition regression;
- finding order and JSON output are deterministic;
- check-state combinations obey §7.1 and operational configuration never
  weakens the normative gate minimum;
- every check that can produce `BLOCK` is completion-required before execution,
  and a completion-optional check never evaluates above `WARN`;
- complete coverage is accepted only for the trusted canonical plan identity,
  and a partial CI projection never uses a complete-gate status identity;
- evaluated findings attach only to completed applicable checks, blocking
  findings and check outcomes agree, and prerequisite graphs are non-empty and
  acyclic where `PREREQUISITE_FAILED` is used;
- malformed or over-budget input never yields a dependent check pass;
- redaction and escaping prevent terminal, JSON, Markdown, and GitHub-command
  injection;
- index, worktree, and commit snapshots never mix content.

Central self-check remains a first-class fixture. The separate dogfood adopter
fixture becomes the first-class self-application parity case.

## 12. Usability and engineering evidence

Record these for the v0.4 baseline and v0.5 beta trials:

| Measure | Why it matters |
|---|---|
| Time to owner-reviewable DRAFT | Separates scaffold speed from the misleading goal of instant green. |
| Time from DRAFT to first trustworthy passing check | Captures the full adoption path. |
| Files and YAML fields manually authored | Tests whether the CLI hides representation work. |
| Profile trigger decisions and unresolved human decisions | Preserves necessary authority instead of hiding it. |
| Terminology lookups or clarification requests | Identifies ontology UX pressure. |
| False/non-actionable warnings | Tests #41 and diagnostic quality. |
| Agent fix/fail loop count | Tests whether authority and next steps are clear. |
| Review-packet questions answered or rewritten by the owner | Measures whether review output is useful. |
| Check runtime and variance on the same repository | Detects performance regressions without inventing a universal threshold. |
| Finding/check-reason-code churn across alpha/beta | Tests whether the diagnostic catalog was frozen too early. |
| Change amplification per rule family | Tests whether the architecture reduces repeated edits. |

For a minimal `core` adoption, target no direct YAML authoring when `init` is
used and no more than the existing essential artifact surface. Do not minimize
the number of genuine human decisions: surfacing those decisions is a product
function. Minimize repeated entry and ambiguity around them.

The G1 report records its selected evidence route, observations, raw counts,
repository/revision, adopter relationship (`internal dogfood`, `affiliated`, or
`external`), operator, decision-maker, decision review class, acceptance cases,
effort cap, and limitations. When structural duplication is the selected route,
it also records the frozen measurement and spike collapse result. It must
distinguish `VERIFIED`, `INFERRED`, and `UNKNOWN` rather than converting two
pilots into a market-wide claim. Missing external evidence remains `UNKNOWN`;
it is not silently replaced by maintainer intuition.

## 13. Release blockers

The following block rc or final:

- a trust-boundary, resource-bound, injection, or fail-closed regression;
- an applicable completion-required check that is omitted, falsely reported
  not-applicable/pass, or left non-completed without a valid §7.1 reason code;
- a check that can produce `BLOCK` in its trusted rule context but is not
  completion-required, or a completion-optional check evaluated above `WARN`;
- a `PREREQUISITE_FAILED` check without a valid `blocked_by` chain, or any
  other non-completed completion-required check without overall
  `INDETERMINATE`;
- complete coverage asserted without the trusted canonical `GatePlanIdentity`,
  or a partial CI projection able to satisfy a complete-gate status identity;
- an unexplained difference from the expected-outcome ledger in finding code,
  check ID/state/reason, `EvaluationKind`, `GateCoverage`, plan identity,
  public dependency edges, evaluated severity, gate effect, or protected
  semantic transition;
- parity evaluated against a ledger or allowlist revision established by the
  implementation change itself rather than an independently accepted revision;
- nondeterministic or non-schema-valid JSON;
- default JSON/text output that leaks restricted content or raw sensitive values;
- migration that changes a stable ID or human decision without explicit review;
- a staged/base/head check that mixes snapshots;
- an installed/pinned engine mismatch that proceeds silently;
- loss of a documented path for v0.4 adopters to remain pinned or migrate;
- a CLI action that can manufacture human review, risk acceptance, or stage
  promotion.

The following do not block v0.5:

- #40 or #41 implementation;
- generic evidence envelope, SARIF, or vendor adapters;
- hosted services or dashboards;
- perfect text wording after the structured contract is correct;
- a full canonical rewrite when G1 says no-go.

## 14. Decisions required before public alpha

Record these as short ADRs:

1. **Product/package/CLI name.** The `AAP` acronym collides with other public
   meanings. Keep the full “OpenDevs Agentic Assurance Profile” name in public
   prose now; verify package and executable namespaces before publishing.
2. **Finding/check-reason shape and allocation.** Codes cannot contain
   severity or stage, cannot be recycled, and need a deprecation policy.
   Finding codes also define versioned fact-key/type schemas.
3. **JSON format and redaction.** Define schema versioning, stable fields,
   location semantics, sensitive-value policy, and deterministic order.
4. **Gate plan and CI status identity.** Define `EvaluationKind`, canonical
   plan manifest/version/digest, trusted plan source and evolution, complete
   and partial status identities, and the names repository rulesets may require.
5. **Adoption-bundle identity and rollout.** Separate it from profile release
   and existing per-document schema versions; define `LEGACY_V0_4`, the first
   explicit identity, context-bound `PRE_V0_4_BASE`, a reserved field namespace
   and collision behavior for today's extension-permitting adoption schema,
   when identity becomes mandatory, and when current/N-1 support and migration
   chaining begin.
6. **Package and profile provenance.** Record the bundled profile commit,
   package/wheel-to-profile linkage, dependency lock and release digest,
   plugin-loading default, installed-source verification, and how a packaged
   CLI detects and explains an engine/profile mismatch.
7. **Git snapshot semantics.** Define worktree, index, commit, merge-base,
   missing-ref, shallow-clone, rename, symlink, and submodule behavior before
   releasing `--staged` or `--base/--head`.
8. **#43 event matrix.** Define `pull_request`, `pull_request_target`,
   `merge_group`, manual, schedule, and reusable-workflow caller behavior
   without introducing privileged execution of untrusted code.
9. **Project root, adoption discovery, and visibility.** Fix root resolution,
   the default declaration path, explicit custom path behavior and trust
   checks, missing-file errors, prohibition on recursive or nearest discovery,
   and the provenance and conservative fallback of repository visibility.

## 15. v0.5 definition of done

The Foundation definition of done is:

- v0.4.0 remains an immutable, reproducible behavioral baseline;
- the Phase 0 evidence record exists and usability claims match its limits;
- every planned check exposes its stable public ID, `completion_required` flag,
  public dependency edges, three §7.1 axes, and state-dependent reason and
  public `blocked_by` data;
- every complete run records a trusted canonical `GatePlanIdentity`, and no
  partial CI projection can satisfy a complete-gate status identity;
- adopter reports expose declared/effective profiles, stage, classification
  review state, and the declared-policy-only meaning of a mechanical pass;
- stable finding, check-reason, and JSON v1 contracts are documented and tested;
- common validation is reachable through one read-only command;
- workflow application logic after trusted checkout lives in tested modules,
  while the minimal pre-checkout trust bootstrap remains explicit;
- any publicly distributed package satisfies the accepted package/profile
  provenance ADR and verifies a mismatch rather than proceeding silently;
- human authority cannot be fabricated by any CLI command;
- an unversioned v0.4 adoption bundle loads through a documented in-memory
  compatibility path after explicit pin upgrade, without mandatory file
  migration;
- rc.1 results match the reviewed ledger across the corpus, security
  boundaries, self-check, dogfood fixture, and both recorded adopter trials;
- deferred issues and known limitations are visible in the release record.

The Progressive UX definition of done, when authorized, additionally requires
preview-first `init`, read-only `review`, explicit unresolved human decisions,
and repeated trials meeting the authorization record's preselected improvement
and non-regression criteria. Without these, the release does not claim the
`init -> check -> review` path.

The Canonical Engine definition of done for `GO(scope)` additionally requires
the model, named rule families, expected-outcome parity, provenance properties,
and any released snapshot-dependent commands within that scope. A full-engine
cutover additionally requires the separate cutover decision and removal of the
legacy implementation from the active v0.5 runtime and test matrix. If G1 is
`NO_GO` or `DEFER`, the decision is recorded and does not hold the Foundation
release hostage.

The intended result is not a larger AAP. It is a smaller public surface over a
deeper, better-bounded core, with honest evidence for whether it is actually
easier to adopt.
